金三银四，工程师简历不要这么写
2021/02/26 · vran

0x00

古语有云：凡事豫（预）则立，不豫（预）则废。

对于跳槽来说也是一样的，就算是金三银四黄金季，如果没有好好的准备也可能会折戟沉沙。

而跳槽首先要准备的自然就是简历，一份好的简历能极大的提高面试邀请率和通过率，使得求职之路事半功倍。

而本文就分享一下写出一份好的简历应该避免的三个误区

  1. 细节太多

  2. 重点太多

  3. 内容太多

PS：由于我自己的职业关系，分享的内容会更偏向于软件工程师一些

0x01 细节太多

作为工程师，在简历中我们经常会用一小节内容来描述自己的技能列表（也叫技术栈），就像下面这样

## 技术栈

1. 熟悉 Java、JavaScript、c、ruby 等语言的字母拼写
2. 熟悉电脑开机、关机等操作
3. ......

而在描述这一节内容时出于以下目的我们往往会过多的添加很多细节性的描述

  1. 提高简历与企业的 JD （Job Description）匹配度
  2. 展现自己的技能深度和广度

在好几年前，我就这么干过

## 技术栈

1. 熟悉 Java、Scala 编程，掌握 OOP 思想，深入研究过 JDK 下部分模块的源码（集合库，并 发库）;
2. 对 Spring 框架部分模块（MVC, IOC, AOP）源码有深入研究，能对其进行一些扩 展定制， 并对其设计有一定的见解 ;
3. 熟悉 Mybatis, Dubbo 等框架，并对其部分源码和设计有一定的阅读研究，能进行 一些简单的扩展定制 ;
4. 对 JVM 有一定的了解，包括常见的一些调优参数，致力于写出 GC 友好的代码 ; 
5. 熟悉常见的设计模式和设计原则，能恰当运用， 追求极简设计 ;
6. 掌握 mysql，redis, memcached, zookeeper 等常见的中间件的使用 ; 
7. 计算机基础扎实，熟悉操作系统原理，常见的数据结构与算法，了解计算机网络协 议等 ; 
8. 爱开源，爱 coding, 魔鬼在细节，认为代码的整体美需要细节来作支撑

这样的内容咋一看很华丽，读过之后却又很难留下什么印象，也就是常说的：一顿操作猛如虎、一看战绩零杠五。

那问题在哪儿呢？

  1. 多条内容其实是重复的，都只是在描述一件事情，比如 1、2、3、4 条其实都只是在像说明熟悉 Java
  2. 空话太多了，缺少信服力，比如第 8 条

那又该如何去调整呢？

  • 将 1、2、3、4 合并为一条，去除细节描述，保留重要关键字
  • 调整第 5 和 第 8 条，增加有信服力的内容

最后优化的内容如下

## 技术栈

1. 熟悉 Java、Scala 以及其对应的框架生态（Spring、Elastic-Job-lite、MyBatis、Dubbo 等）
2. 熟悉主流的设计⽅法、软件架构，主导并负责过多起大型项目的设计
3. 熟悉常用业务中间件：Mysql、Redis、RabbitMq 等
4. 开源项目 PrettyZoo 作者, 多次参与过 Apache 项目 Dubbo 、 Elastic-Job-Lite 的贡献

虽然是以技术栈为切入点进行讨论的，但其实过多的细节描述在项目经验、企业经历等内容中也经常存在。

0x02 重点太多

在简历中通常都会通过加粗或颜色标注的方式来标记某一段关键内容从而引起阅读者的注意。

如果只有一两处标记的话问题不大，但是太多的话反而会让人抓不到重点，形成了反面效果。

0x03 内容太多

对于参加工作多年的同学，一般都会都会有着多份企业经历和项目经验，如果将这些内容都详细的描述在简历上面，那么最后的简历通常都会有两页以上。

然而在实际的面试或筛选中，通常 HR 或面试官比较关注的都是最近做的事情，过多的篇幅只会让简历变得臃肿而已。

一般比较久远的工作经历我建议就直接浓缩成一两行，而最近的工作经历则可以详细描述，比如下面的描述企业就职经历的内容

2017 ~  至今 于 C 企业 \ 任研发总监
1. 负责公司研发团队组织架构、团队文化建设，搭建了百人规模的研发团队
2. 从 0 到 1 搭建了线上营销平台，支撑百万级用户营销，促进线上 DAU 增长 100 倍
3. 负责年度双十一，双十二大促，达成 0 事故指标
2015 ~ 2017 于 B 公司 \ 任 Java 研发工程师 \ 负责电商平台研发工作
2012 ~ 2015 于 A 集团 \ 任 Java 研发工程师 \ 负责集团后端研发工作

对于项目经验来说也是一样的，比较旧远或者没有代表性的项目可以直接略过。

0x04

其实写简历的目标是很清晰的

  1. 通过 HR 或用人部门的筛选
  2. 给面试官留下印象
  3. 让面试官容易找到面试切入点

只要围绕着这些点去实现即可，谨记一个原则： Less is more。
